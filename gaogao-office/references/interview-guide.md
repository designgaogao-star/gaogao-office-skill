# Interview Guide

Use concise numbered questions. Ask only what changes the office design.

## New Project

1. What is this project trying to produce?
2. Who is the primary audience or user?
3. What is the first useful milestone?
4. What kinds of work will happen soon: design, writing, coding, research, QA, migration, release?
5. Which files or areas should be protected from casual edits?

## Existing Project Migration

1. Which old files are authoritative project truth?
2. Should old `vibe/`, planning, or context files be absorbed into the new public office?
3. Should root `AGENTS.md` be replaced after the proposal is reviewed?
4. Should absorbed old framework files be copied only, or moved into `Agent Office/Archive/Legacy Management/` after approval?
5. Which roles should become long-running windows now, and which can wait?

Recommended default: absorb durable facts into `Agent Office/`, copy old framework files into `Agent Office/Archive/Legacy Management/`, move originals only after separate explicit approval, and delete originals only after an exact deletion confirmation.

## Role Planning

Create roles from current work, not a fixed template.

- One role is enough when the first milestone is simple.
- Split a role only when write scope, input, output, or review authority differs.
- Give every role a folder under `Agent Office/Employees/{role-slug}/`.
- Do not create roles that are merely plausible later.
